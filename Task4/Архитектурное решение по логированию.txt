# Архитектурное решение по логированию

## 1. Анализ систем и C4-диаграммы

### 1.1. Компоненты, требующие сбора логов

**Высокий приоритет:**
- **MES API (C#)** — расчёт стоимости, статусы заказов
- **CRM API (SpringBoot)** — подтверждение производства, управление заказами
- **Shop API (SpringBoot)** — приём заказов, загрузка 3D-моделей
- **RabbitMQ** — потеря сообщений, DLQ
- **Базы данных** (MES DB, CRM DB, Shop DB) — критические транзакции

**Средний приоритет:**
- **Frontend** (React, Vue) — действия операторов и менеджеров

---

### 1.2. Список необходимых логов (INFO)

**MES API:**

| Событие | Данные |
|---------|--------|
| Расчёт стоимости | `timestamp`, `order_id`, `calculation_duration_ms`, `polygon_count`, `calculated_price`, `status` |
| Изменение статуса | `timestamp`, `order_id`, `old_status`, `new_status`, `operator_id` |
| Назначение оператору | `timestamp`, `order_id`, `operator_id` |

**CRM API:**

| Событие | Данные |
|---------|--------|
| Создание заказа | `timestamp`, `order_id`, `customer_id`, `source` (web/api) |
| Подтверждение производства | `timestamp`, `order_id`, `manager_id`, `approval_time` |
| Изменение статуса | `timestamp`, `order_id`, `old_status`, `new_status`, `manager_id` |

**Shop API:**

| Событие | Данные |
|---------|--------|
| Создание заказа | `timestamp`, `order_id`, `customer_id` |
| Загрузка 3D-модели | `timestamp`, `order_id`, `file_size`, `upload_duration_ms` |
| Отправка на расчёт | `timestamp`, `order_id`, `target_system` |

**RabbitMQ:**

| Событие | Данные |
|---------|--------|
| Отправка/получение сообщения | `timestamp`, `message_id`, `queue_name`, `order_id` |
| Ошибка доставки | `timestamp`, `message_id`, `queue_name`, `order_id`, `error_type`, `retry_count` |
| Сообщение в DLQ | `timestamp`, `message_id`, `queue_name`, `order_id`, `reason` |

---

### 1.3. Другие уровни логирования

| Уровень | Когда использовать |
|---------|-------------------|
| **DEBUG** | dev/staging: детальная трассировка, отладка интеграций |
| **WARN** | превышение времени обработки (>5 мин), повторные попытки, приближение к лимитам |
| **ERROR** | неудачные операции, ошибки БД/RabbitMQ, потеря сообщений |
| **FATAL** | недоступность сервиса, потеря соединения с БД, компрометация безопасности |

## 2. Мотивация

### Почему нужно логирование

- Потеря заказов и невозможность отследить их путь
- Рост нагрузки (+100 заказов/месяц) и открытие B2B API
- Заказы зависают в непонятных статусах
- Разработчики тратят часы на диагностику

### Что даст

- Снижение MTTR с 2-4 часов до 30 минут
- Сокращение времени диагностики заказа с 20 минут до 2 минут
- Проактивное обнаружение проблем до жалоб клиентов
- Снижение жалоб на 60% и сохранение B2B-контрактов

### Метрики

| Метрика | Текущее | Целевое |
|---------|---------|---------|
| MTTR | 2-4 часа | <30 мин |
| Потеря сообщений в очереди | неизвестно | 0 >1ч |
| Жалобы клиентов | +30%/мес | -60% |
| Время ответа поддержки | 24-48ч | <4ч |

---

### Приоритизация внедрения

**1 этап (критический):**
- RabbitMQ — основная точка потери данных
- MES API — узкое место, долгие расчёты
- CRM API — подтверждение заказов

*Почему сначала эти три: они образуют критический путь заказа, 80% проблем связаны с ними*

**2 этап:** Shop API и базы данных
**3 этап:** Frontend приложения

## 3. Предлагаемое решение

### Стек технологий

- **OpenSearch** — хранение и поиск логов
- **Fluentd** — сбор логов
- **OpenSearch Dashboards** — визуализация

### Архитектура

**1. Applications Layer**
- Shop API (Spring)
- CRM API (Spring)
- MES API (C#)
- RabbitMQ

**2. Collection Layer**
- Fluentd Agents
  - Сбор логов из stdout, файлов, API
  - Парсинг JSON/text
  - Добавление метаданных (service, host, environment)
  - Буферизация и отправка в OpenSearch

**3. Storage & Search Layer**
- OpenSearch Cluster
  - Master Node
  - Data Nodes (2+)
  - Индексы:
    - logs-shop-api-*
    - logs-crm-api-*
    - logs-mes-api-*
    - logs-rabbitmq-*

**4. Visualization Layer**
- OpenSearch Dashboards
  - Поиск и фильтрация логов
  - Дашборды и визуализации
  - Алерты и уведомления
  - Анализ паттернов и аномалий


### Безопасность

- **Доступ по ролям:** Viewer (поддержка), Developer (просмотр + экспорт), Admin (полный доступ)
- **PII не логируются:** ФИО, адреса, телефоны маскируются на уровне Fluentd
- **Аудит** доступа к логам

### Хранение

| Тип | Срок | Назначение |
|-----|------|------------|
| Hot | 7 дней | Оперативный анализ |
| Warm | 30 дней | Периодические запросы |
| Cold | 90 дней | Архивация в S3 |

Отдельный индекс на сервис: `logs-shop-api`, `logs-crm-api`, `logs-mes-api`, `logs-rabbitmq`

### Аналитика и алертинг

**Алерты:**

| Условие | Действие |
|---------|----------|
| Рост ошибок >20% ERROR за 10 мин | Уведомление команде |
| Заказ >3 дней в одном статусе | Создание тикета |
| Нет логов от сервиса >15 мин | Critical alert |

**Аномалии (Anomaly Detection в OpenSearch):**
- Резкий рост RPS (потенциальный DDoS)
- Необычные паттерны ошибок
- Скачок количества заказов от одного клиента

## Дополнительное задание: выбор технологии для логирования

### Критерии выбора

| Критерий | ELK Stack | OpenSearch | Splunk | Grafana Loki |
|----------|-----------|------------|--------|--------------|
| **Лицензия** | Elastic License (ограничения для облачных сервисов) | Apache 2.0 (полностью открытая) | Проприетарная (платная) | Apache 2.0 (открытая) |
| **Производительность индексации** | 50-100K events/sec | 50-100K events/sec | 100K+ events/sec | 20-50K events/sec |
| **Полнотекстовый поиск** | Отлично (Lucene engine) | Отлично (Lucene engine) | Отлично (собственный) | Базовый (Label-based) |
| **Визуализация** | Kibana (богатый функционал) | OpenSearch Dashboards (аналог Kibana) | Splunk Web (мощный, но сложный) | Grafana (отлично для метрик) |
| **ML / Anomaly Detection** | Да (платный в Elastic Stack) | Да (встроенный бесплатно) | Да (встроенный) | Нет |
| **Стоимость владения** | Бесплатно, но дорого при масштабировании | Бесплатно, предсказуемые затраты | Очень дорого ($50K+/год) | Бесплатно, экономично |
| **Поддержка в РФ** | Ограничена | Есть (Yandex Cloud) | Отсутствует | Есть |
| **Экосистема плагинов** | Огромная | Большая (совместим с ELK) | Большая (проприетарная) | Средняя |


Для компании "Александрит" оптимальным выбором является **OpenSearch** по следующим причинам:

1. **Открытая лицензия** — нет рисков для бизнеса
2. **Совместимость** — можно мигрировать с ELK если что-то уже используется
3. **Доступность в Yandex Cloud** — Managed OpenSearch снижает нагрузку на DevOps
4. **Встроенный Anomaly Detection** — критично для выявления DDoS и аномалий
5. **Security из коробки** — RBAC и аудит соответствуют требованиям безопасности
