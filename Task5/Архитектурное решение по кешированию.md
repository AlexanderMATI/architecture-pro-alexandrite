## 2. Мотивация

Внедрение кеширования продиктовано острой необходимостью решения существующих проблем с производительностью, которые напрямую влияют как на внутренних пользователей, так и на клиентов компании.

### Проблемы

1. **Низкая скорость работы операторов в MES** — операторы жалуются на медленную загрузку дашборда со списком заказов. Это напрямую влияет на производительность и, как следствие, на скорость обработки заказов и вознаграждение.

2. **Неудовлетворительная скорость выполнения заказов для новых клиентов** — клиенты, привлекаемые через API, сталкиваются с задержками, что негативно сказывается на репутации и может привести к потере контрактов.

3. **Рост нагрузки на базу данных** — с увеличением числа заказов и подключением новых продавцов через API нагрузка на MES DB постоянно растёт, усугубляя проблемы с производительностью.

### Цели внедрения

- **Ускорить загрузку дашборда MES** — снизить время отклика при отображении списка заказов
- **Повысить общую производительность MES** — ускорить операции чтения данных о заказах
- **Снизить нагрузку на MES DB** — уменьшить количество запросов к базе за счёт обслуживания части запросов из кеша

## 3. Предлагаемое решение

### Тип кеширования: серверное

**Обоснование:**

| Критерий | Почему серверное кеширование |
|----------|------------------------------|
| Централизованное управление | Все инстансы приложений работают с единым источником кешированных данных |
| Согласованность данных | При текущей архитектуре клиентское кеширование создаст проблемы с синхронизацией |
| Контроль инвалидации | Позволяет централизованно управлять актуальностью данных |
| API для партнёров | Клиентское кеширование не применимо для API-интеграций |
| Масштабируемость | При будущем масштабировании не потребуется менять логику кеширования |

**Почему НЕ клиентское кеширование:**
- Невозможно контролировать обновление данных на клиентах
- Не применимо для API-интеграций
- Усложнит синхронизацию при конкурентном доступе операторов
- Не решит проблему нагрузки на БД со стороны API-запросов

---

### Выбор паттернов кеширования

**1. Для дашборда MES (список заказов): Cache-Aside**

| Плюсы | Почему подходит |
|-------|-----------------|
| Простота реализации | Кеш не является критичным — при падении Redis приложение продолжит работу через БД |
| Гибкость | Можно кешировать только часто запрашиваемые данные |
| Ленивая загрузка | Кеш наполняется по мере необходимости |

**Почему НЕ Write-Through:**
- Избыточная сложность для read-heavy сценария
- Операторы преимущественно читают данные, редко изменяют
- Синхронная запись в кеш при каждом обновлении замедлит операции изменения статуса
- Большая часть обновлений приходит через RabbitMQ, что потребует дополнительной интеграции

**Почему НЕ Refresh-Ahead:**
- Невозможно предсказать, какие фильтры и страницы будут запрошены следующими
- Избыточное потребление ресурсов на предварительную загрузку

**2. Для расчёта стоимости: Cache-Aside с долгим TTL**

- Расчёт стоимости для одинаковых 3D-моделей всегда даёт одинаковый результат
- Кеширование по хешу файла модели предотвратит повторные дорогостоящие вычисления

**3. Для API партнёров: Cache-Aside**

- API-запросы могут иметь различные параметры фильтрации
- Нужна гибкость в управлении кешем
- Критично иметь fallback на БД при проблемах с кешем

## 4. Стратегия инвалидации кеша

### Сравнительный анализ стратегий

| Стратегия | Плюсы | Минусы | Применимость |
|-----------|-------|--------|--------------|
| **Программная (по событию)** | Высокая актуальность данных, отсутствие лишних запросов к БД | Требует более сложной логики | **Идеально.** Обеспечивает баланс между производительностью и актуальностью |
| **По времени (TTL)** | Простота реализации | Возможна отдача неактуальных данных; сложно подобрать оптимальный TTL | Менее предпочтительна |
| **По ключу** | Позволяет инвалидировать конкретные части кеша | Усложняет логику | Может быть дополнением к программной инвалидации |

### Выбранная стратегия: комбинированная

| Механизм | Параметры | Назначение |
|----------|-----------|------------|
| **Программная инвалидация** | При изменении статуса заказа | Немедленное обновление кеша после записи в БД |
| **TTL (запасной)** | Список заказов: 60 сек | Страховка на случай сбоя программной инвалидации |
| **LRU Eviction** | maxmemory 2GB | Защита от переполнения памяти |

## 5. Дополнительное задание: сравнение решений

| Критерий | Решение 1: Локальный кеш (MemoryCache) | Решение 2: Внешний распределённый кеш (Redis) |
|----------|----------------------------------------|------------------------------------------------|
| **Описание** | In-memory кеш в рамках инстанса MES API | Отдельный сервис кеширования (Redis) |
| **Диаграмма** | MES API ↔ In-memory Cache | MES API ↔ Redis ↔ MES DB |
| **Плюсы** | Простота внедрения, отсутствие сетевых задержек | Масштабируемость, согласованность между инстансами, кеш не зависит от жизненного цикла приложения, поддержка разных структур данных |
| **Минусы** | Немасштабируемость, несогласованность при нескольких инстансах, потеря кеша при перезапуске | Усложнение архитектуры, сетевые задержки, необходимость администрирования дополнительного сервиса |

### Заключение

**Решение 2 (внешний распределённый кеш на Redis) является более предпочтительным**, несмотря на некоторую сложность внедрения.

**Обоснование:**
- Учитывая планы компании по освоению новых рынков и линейный рост нагрузки, масштабируемость является ключевым фактором
- Использование локального кеша станет "бутылочным горлышком" при горизонтальном масштабировании MES API
- Redis обеспечит согласованность данных между всеми инстансами и станет надёжным и производительным решением в долгосрочной перспективе
