Список существующих и потенциальных проблемных мест систем компании "Александрит":

 1. Отсутствие горизонтального масштабирования — каждый сервис в одном экземпляре, SPOF, деградация при росте нагрузки (+100 заказов/месяц)

 2. Единая БД на запись и чтение — узкое место, блокировки, тормоза дашбордов MES

 3. Ненадёжная обработка сообщений в RabbitMQ — потеря заказов, нет retry и DLQ, жалобы клиентов и партнёров

 4. Синхронный расчёт стоимости (2–30 мин) — таймауты, блокировка, потеря B2B-заказов

 5. Отсутствие observability — нет мониторинга, трейсинга, централизованных логов. Диагностика по жалобам клиентов

 6. Медленные запросы MES — дашборд тормозит даже с пагинацией, операторы не видят новые заказы

 7. Отсутствие кеширования — повторные тяжёлые запросы к БД

 8. Ручное E2E тестирование — релизы блокируются до месяца

 9. Ручной деплой в release/prod — риск ошибок, медленная доставка фиксов

 10. Нет rate limiting на API — внешние партнёры могут перегрузить систему

Разработанные инициативы, необходимые для устранения нежелательных ситуаций

 1. Масштабирование приложений
  - Запустить минимум 2 инстанса каждого сервиса
  - Добавить Load Balancer с health checks
  - Настроить auto-scaling по CPU и RPS

 2. Оптимизация баз данных
  - Master-slave репликация, read replica для отчётов и дашбордов
  - Добавить индексы после анализа slow queries

 3. Асинхронный расчёт стоимости
  - Выделить расчёт в отдельные worker-процессы
  - Перевести на асинхронный режим с уведомлением о готовности
  - Добавить приоритизацию (B2B-заказы выше)

 4. Мониторинг и алертинг
  - Prometheus + Grafana
  - Метрики: latency, traffic, errors, saturation
  - Алерты на ошибки, долгие расчёты, рост очередей

 5. Надёжность RabbitMQ
  - Publisher confirms + consumer acknowledgments
  - Dead Letter Queue для ошибочных сообщений
  - Idempotency keys — защита от дублей

 6. Кеширование
  - Redis для списка заказов и справочников
  - Инвалидация кеша при изменении данных

 7. Оптимизация MES API
  - Найти и убрать N+1 запросы
  - Добавить индексы под критичные запросы
 
 8. Защита API
  - Rate limiting по API-ключу и IP
  - Квоты для B2B-клиентов

 9. Автотесты и CI/CD

  - API-тесты на критичные сценарии (статусы заказа, интеграция)
  - E2E-тесты для основного пути клиента
  - Автоматический деплой в release и prod
  - Blue-green deployment с автоматическим откатом

 10. Долгосрочно: контейнеризация
  - Перевести сервисы в контейнеры
  - Развернуть Kubernetes
  - Service mesh для управления трафиком


Приоритизация инициатив.

1-й месяц: база
  - Мониторинг и алертинг
Без него — вслепую. Нужен для диагностики и подтверждения эффекта от любых изменений.
  - Надёжность RabbitMQ
Прямо влияет на потерю заказов (B2B-контракты под угрозой). 

2-3-й месяц: масштабирование и производительность
  - Горизонтальное масштабирование и read replica
Убирает SPOF, снимает нагрузку с master БД. Без этого система не выдержит дальнейший рост.

  - Кеширование
Решает проблему тормозов дашборда MES после масштабирования.

4-6-й месяц: устойчивость и скорость
  - Асинхронный расчёт стоимости
Устраняет таймауты, позволяет масштабировать обработку независимо.

  - Автотесты и автоматический деплой
Ускоряют релизы, снижают страх изменений, позволяют итеративно развивать систему.

Целевая архитектура через полгода
  - Инфраструктура: 2 или больше инстанса сервисов за LB, PostgreSQL master + read replica, Redis кластер
  - Observability: Prometheus/Grafana (метрики), централизованные логи, трейсинг для отслеживания заказов
  - Надёжность: RabbitMQ с confirm/ack, DLQ, retry; асинхронный расчёт в выделенных worker'ах
  - Производительность: кеширование дашбордов, оптимизированные запросы, индексы
  - Процессы: автотесты в CI, автоматический деплой с быстрым откатом


Топ-3 инициативы на ближайшие полгода

 1. Мониторинг и алертинг 
  - Без него невозможно диагностировать проблемы и оценивать эффект. База для всего остального.

 2. Надёжность RabbitMQ 
  - Прямо останавливает потерю заказов и контрактов. Быстро, с измеримым бизнес-эффектом.

 3. Горизонтальное масштабирование + read replica 
  - Убирает SPOF, готовит систему к росту, снимает нагрузку с БД. Без этого остальные оптимизации не спасут при масштабировании нагрузки.
