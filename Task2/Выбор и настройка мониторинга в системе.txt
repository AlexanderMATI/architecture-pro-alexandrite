1. Мотивация

Сейчас компания узнаёт о проблемах от клиентов — когда заказы уже потеряны, а контракты сорваны. Без мониторинга:

- Не видно, где застревают заказы — на расчёте стоимости, в RabbitMQ или в CRM
- Операторы жалуются на тормоза MES, но нет объективных данных для оптимизации
- Рост нагрузки (+100 заказов/месяц) идёт вслепую — непонятно, когда и что масштабировать
- B2B-партнёры уходят из-за нестабильности API, но мы не можем измерить SLA

Что даст мониторинг:
- Проактивное обнаружение проблем до жалоб клиентов
- Данные для масштабирования (какой сервис упирается)
- Возможность гарантировать SLA партнёрам
- Объективная диагностика вместо «похоже, что тормозит»


2. Выбор подхода к мониторингу

Используем комбинированный подход:

- API сервисов (Shop, CRM, MES): подход RED (Rate, Errors, Duration) — для пользовательского опыта и SLA
- Инфраструктура (БД, RabbitMQ, EC2): подход USE (Utilization, Saturation, Errors) — для выявления узких мест
- Бизнес-процессы: бизнес-метрики — заказы, статусы, SLA


3. Метрики

Response time (latency) for shop API / CRM API / MES API
Зачем: измерение времени ответа API. Для MES это критично из-за жалоб операторов на тормоза дашборда. 
Ярлыки: endpoint (путь API), method (GET/POST), status_code

Number of HTTP 500 for shop API / CRM API / MES API
Зачем: ошибки 500 ведут к потере заказов. Рост 500-х ошибок — первый признак проблем.
Ярлыки: endpoint, method

Number of requests (RPS) for shop API / CRM API / MES API
Зачем: отслеживание нагрузки для планирования масштабирования. Рост RPS без роста ошибок — признак успешного роста бизнеса.
Ярлыки: endpoint

Number of requests (RPS) per user for shop API / CRM API / MES API
Зачем: выявление аномальной активности 
Ярлыки: user_id или api_key

CPU % for shop API / CRM API / MES API
Зачем: выявление узких мест по процессору. При превышении порогов — сигнал к масштабированию.
Ярлыки: instance_id

Memory Utilisation for shop API / CRM API / MES API
Зачем: утечки памяти или недостаточный объём. При >90% риск OOM killer.
Ярлыки: instance_id

Memory Utilisation for shop db instance / MES db instance
Зачем: базы данных — критический компонент. Перегрузка памяти ведёт к замедлению всех запросов.
Ярлыки: database_name

Number of connections for shop db instance / MES db instance
Зачем: при исчерпании пула соединений приложения перестают работать.
Ярлыки: database_name, state (active/idle)

Size of shop db instance / Size of MES db instance
Зачем: планирование дискового пространства. Внезапное заполнение диска = остановка системы.
Ярлыки: database_name

Size of S3 storage
Зачем: отслеживание роста хранилища 3D-моделей. Планирование затрат на облачное хранилище.
Ярлыки: bucket_name


4. План действий

Этап 1. Инфраструктура (1 неделя)
- Развернуть Prometheus (time-series DB) и Grafana на отдельных EC2
- Установить Node Exporter на все инстансы
- Настроить Postgres Exporter для MES DB и Shop DB
- Включить Prometheus plugin в RabbitMQ

Этап 2. Инструментирование приложений (2–3 недели)
- Shop/CRM API (Spring Boot): подключить Micrometer + Prometheus registry
- MES API (C#): подключить prometheus-net
- Добавить кастомные метрики: price_calculation_duration_seconds в MES, order_status_transition в CRM
- Настроить экспорт метрик на /metrics

Этап 3. Дашборды и алерты (1–2 недели)

Дашборды в Grafana:
- Business Overview — заказы по статусам, SLA violations
- MES Performance — время расчёта, дашборд операторов
- Queue Health — размеры очередей, время обработки
- Infrastructure — CPU, память, диски, БД

Алерты в Alertmanager:
- High error rate (>5% 5xx за 5 мин)
- Queue backlog (>1000 сообщений)
- SLA violation (заказ >21 дня)
- CPU >85% >10 мин


5. Дополнительно: показатели насыщенности

CPU >85%: создать тикет, подготовить автоскейлинг
CPU >90%: уведомление DevOps, ручное добавление инстанса

RabbitMQ очередь >1000: тикет на анализ консьюмеров
RabbitMQ очередь >5000: critical alert, масштабирование консьюмеров

Соединения БД >80%: тикет, анализ long-running запросов

Время расчёта >25 мин: логирование
Время расчёта >30 мин: прерывание и уведомление пользователя
